**Identified Failure Patterns & Root Cause:**
Both v1 and v2 STILL classified 100% of queries as [Composite Graphs]. This indicates a persistent bias where the LLM is forced into generating a Composite Graph topology.
The core issues lie in the interaction between stages:
1. **Forced Multiplicity (Stage 1)**: The prompt heavily implies that multiple paths *should* exist for almost every query. When the LLM forces multiple contrived paths for a simple query, it artificially creates a complex graph.
2. **Endpoint Convergence Confusion (Stage 2 & 3)**: If multiple paths to solve a problem exist, they naturally converge at the final answer (the same end node). The previous prompts did not clearly distinguish between sharing *only* the start/end nodes (which should be Divergent) versus sharing *intermediate* nodes (which makes it Composite).
3. **Overly Broad Composite Definition**: By defining Composite as having "convergent endpoints" or just any shared node, almost any multi-path solution to a single question defaults to Composite.

**Prompt Changes Made:**

**v1 & v2 Changes (4-Stage tweaks):**
- `<solution_space_exhaustion>`: 
  - RELAXED the forced multiplicity constraint. Added explicit instructions: "Determine if the query naturally has one path or multiple paths. Do not invent contrived paths if the standard solution is strictly linear."
  - Allowed the LLM to explicitly state "Path A: Direct Resolution" if only one meaningful path exists.
  - Added "Cross-Path Analysis" to explicitly identify if multiple paths share *intermediate* steps, explicitly distinguishing this from sharing *only* the final answer.
- `<logical_supergraph_construction>`:
  - Clarified the NODE UNIQUENESS RULE: Instructed to count whether shared nodes are *intermediate* nodes, or just the unified start/end nodes.
- `<topology_reasoning>`:
  - Completely rewrote the logical definitions to strictly differentiate based on *intermediate* nodes.
  - **[Single Graph]**: Clarified that if Stage 1 yielded only one path, it must be a Single Graph.
  - **[Divergent Graphs]**: Explicitly defined as having multiple methods that share the EXACT same start and end, but have ZERO shared intermediate steps. 
  - **[Composite Graphs]**: Explicitly redefined to require that multiple paths share AT LEAST ONE *intermediate* node before reaching the end.

**Expected Improvement:**
- Maintain 100% parse success rate.
- Break the 100% [Composite Graphs] bias.
- See a natural emergence of [Single Graph] for straightforward queries and [Divergent Graphs] for queries with completely distinct solving methods.
